# Purpose

This Standard Operating Procedure (SOP) defines the standardized process for configuring, validating, maintaining, and monitoring Webhooks in Kommo.

The objective is to ensure that business events generated by Kommo are delivered reliably to external integration platforms, enabling event-driven architectures while maintaining security, traceability, and operational stability.

This SOP establishes best practices for webhook lifecycle management, including registration, validation, monitoring, error handling, and maintenance.

---

# Scope

This procedure applies to any integration that receives business events from Kommo through Webhooks.

Typical use cases include:

- Lead lifecycle events
- Contact lifecycle events
- Pipeline stage changes
- Responsible user changes
- Note creation
- Custom Field updates
- Event-driven synchronization
- CRM automation
- Integration triggers
- Monitoring business activity

This SOP does not cover:

- Webhook consumers implementation
- Business logic processing
- API polling
- OAuth authentication
- Downstream workflow execution

---

# Prerequisites

Before starting this procedure, verify that:

- OAuth authentication has been completed successfully.
- Administrative access to the Kommo account is available.
- The webhook receiver endpoint is operational.
- HTTPS is enabled.
- A valid SSL/TLS certificate is installed.
- Firewall rules allow incoming webhook requests.
- Required business events have been identified.
- Monitoring and logging systems are available.

# Procedure

## 1. Validate Authentication

Verify that:

- Access Token is valid.
- OAuth session is active.
- Administrative permissions are available.

If authentication fails, execute **SOP-KOM-001 OAuth Authentication** before continuing.

---

## 2. Identify Required Events

Determine which business events should trigger webhook notifications.

Common examples include:

- Lead Created
- Lead Updated
- Lead Status Changed
- Contact Created
- Contact Updated
- Note Created
- Responsible User Changed
- Custom Field Updated

Subscribe only to events required by the integration.

---

## 3. Prepare the Webhook Endpoint

Verify that the receiving endpoint:

- Uses HTTPS.
- Is publicly accessible.
- Returns appropriate HTTP responses.
- Accepts POST requests.
- Supports expected payload sizes.
- Processes requests within acceptable response times.

The endpoint should be designed for high availability.

---

## 4. Validate Endpoint Security

Verify that the endpoint implements appropriate security controls.

Recommended controls include:

- HTTPS encryption
- IP allowlists (where applicable)
- Request validation
- Payload validation
- Rate limiting
- Logging
- Replay protection

Do not expose unsecured webhook endpoints.

---

## 5. Register the Webhook

Configure the webhook within Kommo.

Verify:

- Correct callback URL
- Selected business events
- Correct environment
- Proper permissions
- Successful registration

Document the webhook configuration after registration.

---

## 6. Validate Registration

Confirm that:

- Registration completed successfully.
- Callback URL is correct.
- Selected events match business requirements.
- No duplicate registrations exist.

Reject incomplete or incorrect configurations.

---

## 7. Trigger Test Events

Generate controlled business events to verify webhook delivery.

Examples include:

- Update a Lead
- Update a Contact
- Create a Note
- Modify a Custom Field

Testing should be performed in a non-production environment whenever possible.

---

## 8. Validate Incoming Requests

Verify that received webhook requests contain:

- Expected HTTP method
- Valid headers
- Valid payload
- Expected event type
- Correct entity identifiers
- Required timestamps

Reject malformed or incomplete requests.

---

## 9. Validate Event Processing

Confirm that:

- Events are processed successfully.
- Payload structure matches expectations.
- Entity identifiers are valid.
- Downstream workflows execute correctly.
- Duplicate events are handled safely.

Webhook receipt does not guarantee successful business processing.

---

## 10. Implement Idempotency

Ensure repeated delivery of the same event does not produce duplicate business actions.

Recommended techniques include:

- Event identifier tracking
- Entity version comparison
- Timestamp validation
- Processing history
- Duplicate detection

Every webhook consumer should be designed to be idempotent.

---

## 11. Monitor Webhook Health

Continuously monitor:

- Delivery success rate
- Response times
- HTTP error rates
- Processing failures
- Duplicate deliveries
- Endpoint availability
- Unexpected payloads

Monitoring should support proactive incident detection.

---

## 12. Record Operational Information

Record webhook activity.

Recommended information includes:

- Event type
- Entity ID
- Delivery timestamp
- Processing status
- HTTP response
- Execution duration
- Integration identifier

Sensitive customer information should not be unnecessarily logged.

---

## 13. Return the Result

Return an appropriate HTTP response to Kommo.

Successful processing should acknowledge receipt promptly.

Business processing should not unnecessarily delay the webhook response.

---

# Validation

The procedure is considered successful when:

- Authentication succeeded.
- Webhook registration completed successfully.
- Endpoint is reachable.
- Test events were delivered.
- Payload validation succeeded.
- Business events were processed correctly.
- Monitoring confirms successful operation.

---

# Expected Result

Upon successful completion:

- Kommo delivers subscribed events to the configured endpoint.
- Business events are received reliably.
- Payloads are validated.
- Duplicate processing is prevented.
- Operational monitoring is active.
- The integration operates using an event-driven architecture.

# Common Errors

| Error | Possible Cause | Resolution |
|---------|----------------|------------|
| Unauthorized (401) | Invalid OAuth credentials | Refresh authentication |
| Forbidden (403) | Missing permissions | Review OAuth scopes |
| Webhook Not Triggered | Event not subscribed | Verify event configuration |
| Endpoint Unreachable | Network or DNS issue | Verify endpoint availability |
| SSL Validation Failed | Invalid certificate | Install valid TLS certificate |
| HTTP 404 | Incorrect callback URL | Verify endpoint configuration |
| HTTP 500 | Internal processing failure | Review application logs |
| Timeout | Slow endpoint response | Optimize processing |
| Duplicate Events | Retry delivery | Implement idempotency |
| Invalid Payload | Payload validation failed | Review webhook schema |

---

# Troubleshooting

| Issue | Diagnostic Action | Resolution |
|--------|-------------------|------------|
| No webhook received | Verify registration and subscribed events | Correct configuration |
| Endpoint returns errors | Review application logs | Resolve processing issue |
| Payload validation fails | Compare payload with API specification | Update parser |
| Duplicate processing | Inspect idempotency implementation | Strengthen duplicate detection |
| Slow processing | Measure execution time | Optimize processing logic |
| Authentication errors | Validate OAuth session | Re-authenticate |
| SSL problems | Verify certificate chain | Install valid certificate |
| Unexpected events | Review subscriptions | Remove unnecessary event types |

---

# Rollback

If the webhook configuration cannot be completed successfully:

1. Disable the affected webhook.
2. Record the incident in operational logs.
3. Verify endpoint availability.
4. Correct configuration errors.
5. Re-register the webhook if necessary.
6. Execute validation tests before re-enabling production traffic.

If an incorrect webhook configuration has already been deployed:

1. Remove the incorrect registration.
2. Restore the previous configuration.
3. Verify successful event delivery.
4. Confirm downstream integrations remain operational.
5. Record the corrective action.

---

# Best Practices

- Prefer event-driven integrations over polling whenever possible.
- Subscribe only to required business events.
- Use HTTPS for every webhook endpoint.
- Validate every incoming request.
- Implement idempotent processing.
- Respond quickly to webhook requests.
- Perform business processing asynchronously when appropriate.
- Log operational events without exposing sensitive data.
- Monitor webhook delivery continuously.
- Maintain separate webhook endpoints for different environments.
- Regularly review webhook subscriptions.
- Periodically validate endpoint availability and security.
